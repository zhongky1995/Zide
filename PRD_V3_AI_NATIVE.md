## 1. 产品机会与目标

### 1.1 背景与机会

在方案、报告、汇报这类功能型长文上，通用 AI 已经能提供相当高的基础质量，用户未必需要一个专门产品来替代现有工作流。

但在长篇小说创作上，通用 AI 仍然有明显短板：

1. 世界观和设定容易漂移。
2. 人物说话方式和行为动机不稳定。
3. 长线剧情推进容易断裂。
4. 伏笔、时间线、角色关系难以持续维护。
5. 中后期返工成本极高，作者容易弃坑。

因此，Zide 更适合把产品楔子切入到：

`长篇小说作者的持续创作与完稿系统`

而不是继续把自己定位成通用长文 AI 工作台。

### 1.2 用户价值一句话

把小说创作从“边写边忘、越写越乱”变成“设定可记、剧情可推、问题可审、成稿可收口”的持续创作流程。

### 1.3 业务目标一句话

提升长篇小说项目的持续推进率、设定一致性和完稿率。

### 1.4 本期要解决的核心矛盾

1. 作者知道世界很大，但系统帮不了他稳住世界。
2. 作者知道人物该怎么走，但中长篇里容易写偏。
3. 作者知道剧情要推进，但不知道下一场该写什么最关键。
4. AI 会写局部场景，但不会对整部长篇负责。
5. 返工常常发生在几十章之后，修起来代价极高。

### 1.5 本期不做范围

1. 不做多人实时协作写作。
2. 不做社区、连载平台和读者互动系统。
3. 不做 IP 交易或模板市场。
4. 不做专业排版出版流程。
5. 不做移动端优先，仍以桌面端为主。

## 2. 用户身份与场景锚点

### 2.1 主用户身份卡

**主用户：长篇小说作者**

1. 典型角色：网文作者、独立小说作者、同人/长篇故事创作者。
2. 核心目标：持续推进一部长篇作品，保持世界观、人物、剧情和文风稳定，并最终完成可连载或可完稿的稿件。
3. 当前能力：有创作冲动和基础写作能力，会用 AI 辅助，但不想自己手工维护大量设定文档和时间线表格。
4. 核心顾虑：
   - 写到中后期设定冲突
   - 人物 OOC
   - 剧情推进失速
   - 伏笔埋了忘收
   - 重写代价过高

### 2.2 次级用户与兼容角色

1. 轻小说/剧本式创作者：更看重节奏、冲突和分幕结构。
2. 叙事策划/剧情设计者：更看重设定系统化和角色弧线。
3. 编辑/审稿者：更看重一致性问题、剧情缺口和版本回退。

### 2.3 主场景描述

**主场景：长篇小说从灵感到连载/完稿的持续推进**

1. 触发条件：作者有一个故事种子，希望把它发展为长篇小说。
2. 环境：桌面创作场景，长时间单人写作。
3. 时间压力：中等，但持续周期长。
4. 设备：Mac/Windows 桌面端。
5. 决策特点：单次写不好可以改，但最怕“长期失控”和“后期大返工”。

### 2.4 次要场景与边界

1. 从一个故事设定快速展开世界观和角色群像。
2. 以卷/幕/章节方式规划长线剧情。
3. 写到中期后进行连续性审查和补漏。
4. 进行 retcon 后评估受影响章节。

### 2.5 现有替代方案与痛点

| 替代方案 | 用户为什么会用 | 主要痛点 |
|---|---|---|
| 通用对话式 AI | 方便、上手快 | 写得快但记不住，越长越漂 |
| 文档编辑器 + 笔记软件 | 可控 | 世界观、角色、时间线分散，维护成本高 |
| 专门写作软件 | 有大纲和卡片 | AI 推进能力弱，连续性审查弱 |
| 现有 Zide | 有项目化骨架 | 还像通用长文工具，不像小说创作系统 |

### 3.1 当前旅程地图

| 阶段 | 触发/场景 | 用户目标 | 用户动作 | 接触点 | 情绪/顾虑 | 痛点 | 机会点 | 产品能力/设计要求 |
|---|---|---|---|---|---|---|---|---|
| 灵感进入 | 有一个故事念头 | 先把核心火花留下来 | 新建项目、写设定草稿 | 项目创建页 | 兴奋又模糊 | 灵感容易散 | 系统先帮作者收束故事核 | 故事种子输入、主题提炼 |

| 完稿收束 | 准备连载或完稿 | 确认能否继续发/是否该总修 | 导出、回退、查看问题 | 成稿中心 | 紧张 | 不知道现在是草稿还是可发状态 | 提供成稿准备度和缺口提示 | Manuscript Readiness |
| 复盘复用 | 进入下个阶段 | 复用设定和经验 | 查看记忆资产 | 记忆中心 | 希望复用 | 历史决定找不到 | 让历史资产结构化沉淀 | Lore Memory Center |

### 3.2 目标旅程地图

| 阶段 | 目标体验 | 用户看到什么 | 系统承担什么 |
|---|---|---|---|
| 灵感进入 | 用户先讲故事，系统帮忙成型 | 故事种子、主题、核心冲突草案 | Seed Router + Story Planner |
| 设定沉淀 | 世界观、人设和风格持续可用 | Story Bible、角色卡、规则卡 | Context System + Memory |
| 剧情架构 | 大纲变成剧情推进骨架 | 卷/幕/章节目标、伏笔标记、节奏风险 | Planner + Plot Generator |
| 场景写作 | 用户选任务，系统选方法 | 快速润色 / 推进场景 / 深改剧情 | Task Router + Generator |
| 连续性审查 | 结果先审再采纳 | 候选稿、设定冲突、人物偏移、时间线问题 | Critique Loop |
| 完稿收束 | 作者知道当前稿件是否可连载/可完稿 | 成稿准备度、阻塞问题、导出建议 | Acceptance Gate + Manuscript Center |
| 复盘复用 | 系统记住稳定设定和剧情决议 | Lore 卡片、角色摘要、时间线摘要 | Long-term Memory |

### 3.3 关键机会点

1. 让系统主动维护世界观和角色一致性。
2. 让 AI 输出先进入候选稿，不直接污染正文。
3. 把“设定冲突、人物失真、剧情断裂”变成产品级检查。
4. 把 retcon 设计成系统能力，而不是手工返工。
5. 把长篇写作的“持续推进感”做出来，而不是只做局部生成。

### 3.4 产品体验目标

1. 更稳的设定记忆。
2. 更顺的剧情推进。
3. 更少的中后期崩盘。
4. 更强的问题可见性。
5. 更高的完稿可能性。

### 3.5 体验设计原则

1. 先保一致性，再追求局部华丽。
2. 用户决定创作方向，系统负责维护连续性。
3. 所有高风险改动都要可回退。
4. 长期记忆必须可见、可纠正。
5. 系统默认帮助作者继续写，而不是让作者重新组织一堆工具。

## 4. 需求结构与优先级

### 4.1 需求清单

#### P0：本期必须

1. Story Bible Studio
   - 管理故事种子、主题、世界观、角色、规则、文风
2. Plot Board
   - 支持卷/幕/章节/剧情节拍/伏笔标记
3. Scene Sprint
   - 用任务型入口推进场景写作
4. 候选稿机制
   - AI 结果先进入候选稿，不直接写正式正文
5. Continuity Review
   - 检查设定冲突、人物偏移、时间线问题、伏笔漏收
6. Lore Memory Center
   - 展示稳定设定、角色摘要、关系和时间线摘要
7. Manuscript Center
   - 展示成稿准备度、章节状态、导出与快照
8. Run Console
   - 展示 route、plan、critique、revision 轨迹

#### P1：增强版

1. 多主角/多 POV 管理。
2. 角色关系图谱。
3. 节奏与张力曲线提示。
4. retcon 影响范围分析。
5. 连载包导出。

#### P2：扩展版

1. 共同创作与审稿。
2. 类型小说模板和写作工坊。
3. 角色语音/语气特征更精细的模型控制。

### 4.2 需求来源映射

| 需求 | 来源阶段 | 对应痛点 |
|---|---|---|
| Story Bible Studio | 设定沉淀 | 世界观和人设分散，后续容易忘 |
| Plot Board | 剧情架构 | 大纲像标题列表，无法支撑长线推进 |
| Scene Sprint | 场景写作 | 作者卡文，不知道下一场怎么推进 |
| Continuity Review | 连续性审查 | 设定冲突、人物偏移、时间线错乱 |
| Lore Memory Center | 复盘复用 | 稳定信息无法沉淀成长期资产 |
| Manuscript Center | 完稿收束 | 不知道现在能否继续连载或总修 |

### 4.3 优先级依据

1. 是否直接提升长篇持续推进率。
2. 是否能降低中后期返工成本。
3. 是否能体现相对通用 AI 的差异化优势。
4. 是否能为 AI Native 内核落地提供真实场景。

### 4.4 MVP 范围

本期 MVP 只服务一个主锚点：

`长篇小说作者在桌面端持续推进一部长篇作品`

MVP 包含：

1. Story Bible Studio
2. Plot Board
3. Scene Sprint
4. 候选稿 + Continuity Review
5. Lore Memory Center 基础版
6. Manuscript Center 基础版
7. Run Console 基础版

### 4.5 非 MVP 范围

1. 协作写作。
2. 社区连载。
3. 专业出版排版。
4. 多端同步。

## 5. 产品架构设计

### 5.1 功能架构

| 模块 | 服务对象 | 核心职责 | 上下游关系 |
|---|---|---|---|
| Story Bible Studio | 项目启动作者 | 收敛故事种子、世界观、角色、规则、文风 | 上游：项目创建；下游：Plot Board |
| Plot Board | 剧情规划作者 | 生成卷/幕/章节/节拍和伏笔规划 | 上游：Story Bible；下游：Scene Sprint |
| Scene Sprint | 场景写作作者 | 以任务方式推进当前场景或章节 | 上游：Plot Board；下游：Continuity Review |
| Continuity Review | 修订作者 | 显示连续性问题、修复建议、采纳决策 | 上游：候选稿；下游：正式正文与 Manuscript Center |
| Lore Memory Center | 全部作者 | 展示长期稳定设定、角色和时间线记忆 | 支撑 Story Bible / Scene / Review |
| Manuscript Center | 收束阶段作者 | 展示成稿准备度、章节完成状态、导出和快照 | 上游：Review；下游：导出结果 |
| Run Console | 想理解 AI 过程的作者 | 展示每次任务的执行轨迹 | 贯穿所有模块 |

### 5.2 信息架构

#### 页面导航

```text
│   ├── 下一步建议任务
│   ├── 成稿准备度摘要
│   └── 最近运行记录
├── Story Bible Studio
├── Plot Board
├── Scene Sprint
├── Continuity Review
├── Lore Memory Center
├── Manuscript Center
└── Run Console
```

#### 页面职责

| 页面 | 页面职责 | 关键组件 |
|---|---|---|
| Overview | 告诉作者现在最该写哪里、项目是否稳定 | 写作健康卡、下一步任务卡、准备度卡 |
| Story Bible Studio | 建立作品基础设定 | 故事种子、世界观卡、角色卡、规则卡 |
| Plot Board | 管理剧情推进骨架 | 卷/幕/章节树、节拍卡、伏笔标记 |
| Scene Sprint | 当前写作主战场 | 正式正文区、候选稿区、任务入口、章节导航 |
| Continuity Review | 一致性问题定位与修复 | 评分卡、设定冲突列表、人物偏移列表、时间线问题 |

```text
  ├── Plot Board
```

### 5.3 关键业务规则

1. 所有 AI 任务都先形成一个 `Task Run`。
2. AI 结果默认进入 `候选稿`，不直接覆盖正式正文。
3. 只有通过系统门禁或作者显式采纳，候选稿才能进入正式正文。
4. 所有长期记忆只写入作者已确认的稳定信息。
5. 作者进行 retcon 时，系统必须记录：
   - 变更内容
   - 受影响角色/设定/章节
   - 是否需要触发连续性复查
6. Continuity Review 是主流程，不是边缘工具页。
7. Manuscript Center 不只负责导出，还要给出当前作品的成稿准备度。

## 6. 核心流程与交互

### 6.1 主流程

1. 作者输入故事种子或创作想法。
2. 系统在 Story Bible Studio 中生成世界观、角色、主题和规则草案。
3. 作者确认核心设定后进入 Plot Board。
4. 系统生成卷/幕/章节骨架、节拍和伏笔建议。
5. 作者确认剧情骨架后进入 Scene Sprint。
6. 作者对当前章节/场景提交任务型指令：
   - 快速润色
   - 推进场景
   - 深改剧情
7. 系统执行任务并产出候选稿。
8. 系统自动进行连续性审查。
9. 作者在 Continuity Review 中选择：
   - 继续修正
   - 采纳候选稿
   - 放弃结果
10. 系统把已确认的稳定信息更新到 Lore Memory Center。
11. 作者在 Manuscript Center 查看作品准备度并导出当前稿件。

### 6.2 异常流程 A：候选稿触发设定冲突

1. 系统生成候选稿。
2. Continuity Review 判定与既有设定冲突。
3. 系统不给正式正文写入权限，只显示候选稿和冲突点。
4. 作者可选择修正当前候选稿，或确认这是一次 retcon。
5. 若确认 retcon，系统记录变更并提示受影响章节。

### 6.3 异常流程 B：人物出现明显 OOC

1. 系统标记候选稿中人物行为或语气偏离既有人设。
2. 作者可查看对比说明。
3. 作者可选择：
   - 回修到符合人设
   - 更新角色卡，承认人物发生变化
4. 若更新角色卡，系统将该变化写入长期记忆并提示后续章节。

### 6.4 异常流程 C：剧情推进失速

1. 作者多次尝试生成但仍无可采纳结果。
2. 系统提示当前章节可能存在剧情目标不清或节拍断裂。
3. 作者可一键跳回 Plot Board 调整章节目标或节拍。
4. 调整完成后再回到 Scene Sprint 继续写作。

### 6.5 关键状态、反馈与恢复机制

| 状态 | 作者看到什么 | 可执行动作 |
|---|---|---|
| 任务执行中 | 当前 route、当前步骤、预计下一步 | 取消、等待、继续 |
| 生成完成待评审 | 候选稿、连续性评分、问题摘要 | 查看详情、继续修、采纳 |
| 连续性高风险 | 设定冲突、OOC、时间线问题 | 修正、声明 retcon、放弃 |
| 成稿准备度不足 | 阻塞问题、薄弱章节、未收伏笔 | 跳转修复、延后导出 |
| 已导出 | 当前稿件版本、快照、导出记录 | 打开目录、回退、继续写作 |

## 7. 页面级交互细化

### 7.1 Overview

目标：让作者打开项目时，第一眼知道“故事现在卡在哪、下一步最该写什么”。

核心卡片：

1. 当前剧情推进卡
2. 下一步建议任务卡
3. 成稿准备度卡
4. 最近 Task Run 卡

### 7.2 Story Bible Studio

目标：把故事想法收敛成稳定的创作底座。

核心区域：

1. 故事种子输入区
2. 世界观区
3. 角色卡区
4. 风格与叙事语气区
5. 稳定规则确认区

### 7.3 Plot Board

目标：让剧情结构从“灵感点子”升级为“可持续推进的骨架”。

核心区域：

1. 卷/幕/章节树
2. 每章目标面板
3. 节拍与冲突卡
4. 伏笔与回收标记
5. 进入 Scene Sprint 的章节入口

### 7.4 Scene Sprint

目标：让作者围绕当前场景推进，而不是在功能之间切换。

核心区域：

1. 正式正文区
2. 候选稿区
3. 任务型入口区
4. 当前章节目标与上下文摘要区
5. 跳转 Continuity Review / Run Console 的快捷入口

### 7.5 Continuity Review

目标：让一致性问题、修复动作和采纳决策收口在一个主页面。

核心区域：

1. 连续性评分卡
2. 设定冲突列表
3. 人物偏移列表
4. 时间线问题列表
5. 修复建议与 retcon 入口

### 7.6 Lore Memory Center

目标：让作者知道系统记住了什么，以及这些记忆是否可信。

核心区域：

1. 角色卡
2. 世界规则卡
3. 时间线卡
4. 关系卡
5. 已确认剧情决策卡

### 7.7 Manuscript Center

目标：从“能导出”升级为“知道当前稿件是否适合继续连载或收口总修”。

核心区域：

1. 成稿准备度
2. 章节状态
3. 主要阻塞问题
4. 导出记录
5. 快照管理

### 7.8 Run Console

目标：把 AI 的黑盒过程变成作者可理解的过程。

核心区域：

1. route 决策
2. plan 步骤
3. critique 记录
4. revision 轮次
5. 最终采纳结果

## 9. 指标与验证

### 9.1 北极星指标

`项目持续推进到可完稿状态的成功率`

定义：一部小说项目在系统内经过持续创作后，达到作者认可的“可继续连载或进入总修/完稿”的比例。

### 9.2 过程指标

1. Scene Task Completion Rate
2. First Pass Continuity Pass Rate
3. Average Revision Rounds
4. Candidate-to-Accepted 转化率
5. Continuity Review Coverage
6. Plot Stall Ratio

### 9.3 守护指标

1. 强制采纳高风险候选稿比例不能持续过高。
2. retcon 频率不能持续失控。
3. 手工完全重写比例不能持续过高。
4. 中途弃坑率不能持续上升。

### 9.4 数据观察周期与验收标准

1. 周观察：任务完成率、连续性通过率、修正轮次。
2. 月观察：完稿推进率、弃坑率、retcon 率。

本期验收建议：

1. 至少 80% 的 AI 场景任务进入 Continuity Review 覆盖。
2. 至少 60% 的任务能在 2 轮内达成可采纳结果。
3. 作者对“设定记忆和角色稳定性”主观满意度明显高于旧版本。

## 10. 风险与对策

### 风险 1：系统过度保守，写出来不够有创作感

应对：

1. 区分“稳定推进”和“创意探索”两类任务路径。
2. 先把连续性门禁用于候选稿阶段，不直接扼杀探索。
3. 允许作者手动采纳高风险结果，但必须留痕。

### 风险 2：设定维护复杂，反而增加作者负担

应对：

1. Story Bible 以 AI 草案为起点，不要求作者手工填满。
2. 长期记忆只展示稳定资产，不展示全部噪音。
3. 关键设定支持作者快速确认和修订。

### 风险 3：retcon 处理不好会导致全局更乱

应对：

1. retcon 必须结构化记录。
2. retcon 后提示受影响章节。
3. 高风险 retcon 后默认触发连续性复查。

### 风险 4：从旧产品迁移时割裂感强

应对：

1. 保留现有章节编辑能力。
2. 用灰度方式把按钮入口迁移成 Scene Sprint 任务入口。
3. 保留旧大纲和导出能力作为兼容层。
